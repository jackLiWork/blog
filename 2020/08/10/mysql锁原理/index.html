mysql中锁

意向锁：具有表级粒度，主要用于在对某个表加表锁时判断此表是否已存在行级锁，因此意向共享锁和意向排他锁之间是相互兼容的，因此在加表锁或行锁前都要先获取对应意向锁。

索引记录锁：行级锁，但是锁是加在索引上，分为聚族索引锁和非聚族索引锁（非聚族索引锁最终都会走到聚族索引，因此聚族索引一定会被加锁）
间隙锁：仅会加在用到的索引间隙（即若使用非聚族索引将不会在聚族索引上加间隙锁），仅事务隔离级别大于等于RR时使用，主要用于解决幻读。因此间隙锁之间是相互兼容的；但是持有间隙锁时，请求获取插入意向锁会被阻塞。
Next-key锁：为索引记录锁+间隙锁组合，间隙为当前记录之前的间隙，左开右闭，InnoDB存储引擎还会对辅助索引下一个键值加上gap lock。
插入意向锁：为特殊的间隙锁，仅事务隔离级别大于等于RR时使用，主要用于解决并发插入的性能。当插入新数据获取其他锁之前，需要先获取插入意向锁。插入意向锁是互相兼容的，也无所谓共享或排斥。当同一间隙存在间隙锁时，插入意向锁会被阻塞；但当同一间隙存在插入意向锁时，间隙锁可以添加成功。

加锁操作

SELECT … FROM… 语句：当事务隔离级别为RR时，不会加锁，通过MVCC解决可重复读
SELECT … FOR UPDATE or LOCK IN SHARE MODE, UPDATE, DELETE语句：当事务隔离级别为RR时，对于使用唯一索引和唯一搜索条件（即搜索到记录为离散且与索引一一对应），仅加行锁；对于非唯一索引或其他搜索条件（包括搜索不到情况），会对搜索过程中遇到的记录加next-key锁，若记录不存在则会加间隙锁。
INSERT语句：在插入记录所在的聚族索引或唯一索引上设置索引记录排他锁，并且在设置索引记录排他锁前会先获取插入意向锁。当简单的insert语句当遇到唯一键冲突时，会对索引记录请求加共享记录锁。当insert … on update …语句当遇到唯一键冲突时，会对索引记录请求加排他next-key锁。

注意加锁是在搜索过程中对遇到的索引记录加锁，因此对于无索引的搜索条件将会对全表所有记录的聚族索引加next-key锁

死锁日志

tables in use 1 表示有一个表被使用，locked 1 表示有一个表锁。LOCK WAIT 表示事务正在等待锁，4 lock struct(s) 表示该事务的锁链表的长度为 4，每个链表节点代表该事务持有的一个锁结构，包括表锁，记录锁以及 autoinc 锁等。heap size 1136 为事务分配的锁堆内存大小。
12 row lock(s) 表示当前事务持有的行锁个数。undo log entries 1 表示当前事务有 1 个 undo log 记录，因为二级索引不记 undo log，说明该事务已经更新了 1 条聚集索引记录。

这里显示的是正在等待锁的 SQL 语句，死锁日志里每个事务都只显示一条 SQL 语句，这对我们分析死锁很不方便，我们必须要结合应用程序去具体分析这个 SQL 之前还执行了哪些其他的 SQL 语句，或者根据 binlog 也可以大致找到一个事务执行的 SQL 语句。

RECORD LOCKS 表示记录锁（并且可以看出要加锁的索引为 order_id_un），space id 为 300，page no 为 5480，n bits 552 表示这个记录锁结构上留有 552 个 bit 位（该 page 上的记录数 + 64）。lock_mode X 表示该记录锁为排他锁，insert intention waiting 表示要加的锁为插入意向锁，并处于锁等待状态。

目前有四种类型的行锁：记录锁，间隙锁，Next-key锁和插入意向锁。这四种锁对应的死锁日志各不相同，如下：

  • 记录锁（LOCK_REC_NOT_GAP）: lock_mode X locks rec but not gap
  • 间隙锁（LOCK_GAP）: lock_mode X locks gap before rec
  • Next-key 锁（LOCK_ORNIDARY）: lock_mode X
  • 插入意向锁（LOCK_INSERT_INTENTION）: lock_mode X locks gap before rec insert intention

注意如果是在 supremum record 上加锁，locks gap before rec 会省略掉，间隙锁会显示成 lock_mode X，插入意向锁会显示成 lock_mode X insert intention。如下所示：

heap no 1表示这个记录是 supremum record；相应的，heap no 0表示这个记录是infimum record 的 ）。

HOLDS THE LOCK(S)，表示事务二持有什么锁，这个锁往往就是事务一处于锁等待的原因。这里可以看到事务二正在等待索引 order_id_un 上的插入意向锁，并且它已经持有了一个 X 锁（Next-key 锁，也有可能是 supremum 上的间隙锁）。

当看到死锁日志时，可以对死锁原因有一个大概判断，但是对死锁的诊断还需要结合事务执行SQL语句的上下文，从而理清每个事务的 SQL 执行顺序，更有利于分析出死锁的真正原因。
