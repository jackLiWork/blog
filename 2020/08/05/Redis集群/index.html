Redis集群

2020-08-05 Redis学习 Redis, Redis集群 评论

简介

Redis集群是Redis提供的分布式数据库方案，集群通过分片来进行数据共享，并提供复制和故障转移功能。集群中存在福多个节点，每个节点都由多台机器组成的一主多从结构的系统。

集群的启动

集群中节点的启动过程和单机模式下服务器的启动过程相同，但是会有一些额外的逻辑用于处理集群相关逻辑，如时间事件处理器中增加clusterCron函数，用于发送Gossip消息、检查节点状态、或者节点下线时执行故障转移操作。
在集群刚启动时，集群中每个节点都是相互独立的，各个节点间可以通过发送CLUSTER MEET 命令实现互联，具体的互联步骤如下：

  1. 通过客户端向节点A发送命令CLUSTER MEET <B.ip> <B.port>

  2. 收到此命令的节点A会首先为节点B创建clusterNode结构对象，并将该结构添加到自己的clusterState.nodes字典中

  3. 节点A将根据CLUSTER MEET命令中给定的IP和端口号，向节点B发送MEET消息

  4. 当节点B收到节点A发送的MEET消息时，节点B也会为节点A创建一个clusterNode结构对象，并将该结构对象添加到自己的clusterState.nodes字典中，然后向节点A返回一条PONG消息

  5. 节点A收到节点B返回的PONG消息后，节点A会向节点B再返回一条PING消息，表示握手完成

集群的分片

Redis集群通过分片的方式来保存数据库中的键值对，集群的整个数据库被分为16384个槽，对于每个key，客户端都要先经过CRC16(key)%16384计算，来获取此key放在哪个槽位上，然后再根据客户端记录的槽与节点地址（通常为节点中主机地址）的映射关系缓存（初始是通过slots命令来获取全部映射关系的）来找到对应的服务器地址（若为写命令，则为主机服务器的地址；若为读命令则再根据主从映射缓存找到从机服务器的地址），再将此命令发送给对应的服务器。

通过客户端向节点A发送CLUSTER ADDSLOTS [slots]命令，可以将一个或多个槽指派给对应的节点A，在clusterNode结构对象中的slots位数组中记录节点负责的槽位。如下图所示：

集群的重新分片

Redis集群的重新分片操作可以将某个节点上（源节点）负责的任意数量的槽迁移到其他节点（目标节点）进行负责（注意与槽相关的键值数据也会一并迁移到目标节点中），并且支持在线迁移。因此Redis集群的重新分片操作主要用于数据迁移。

Redis集群分片是通过集群管理软件redis-trib负责执行的，redis-trib对集群的单个槽slot进行重新分片的步骤（从节点A将槽slot迁移到节点B）如下：

  1. redis-trib向目标节点B发送命令CLUSTER SETSLOT IMPORTING <节点A的source_id>，目标节点B接收到此命令后，会将clusterState.import_slots_from[slot]指针指向节点B中记录节点A状态的clusterState结构对象，表示节点B已准备好接收由节点A迁移过来的槽slot及所属键值对数据

  2. redis-trib向源节点A发送命令CLUSTER SETSLOT MIGRATING ，源节点A接收到此命令后，会将clusterState.migrating_slots_to[slot]指针指向节点A中记录节点B状态的clusterState结构对象，表示节点A负责的槽slot及所属键值对数据将会被迁移到节点B

  3. redis-trib向源节点A发送命令CLUSTER GETKEYSINSLOT ，源节点A接收到此命令后会将属于此槽slot下的最多count个键值对返回给redis-trib

  4. redis-trib遍历步骤三中源节点A返回的键值对，对于每个键值对redis-trib都会向源节点A发送命令MIGRATE <0> ，源节点A在收到此命令后会将命令中的键对应的键值对原子性的迁移到目标节点B中

  5. 重复步骤3和步骤4，直到源节点A上属于槽slot的键值对都迁移到目标节点B上。当迁移完成后，redis-trib会向集群中任一节点发送CLUSTER SETSLOT NODE 命令，通知此节点槽slot已经被指派给目标节点B，最终通过gossip协议将保证集群中所有节点知道槽slot已经被指派给目标节点B

我们知道由于集群支持在线槽迁移，因此在迁移过程中，肯定会存在一个槽的键值对数据位于两个节点上，此时集群提供了ASK错误来指引客户端转向迁移后的节点上去执行命令。假设槽id正在从节点A迁移到节点B，具体的步骤如下：

  1. 客户端向节点A发送命令去执行与key有关的操作
  2. 节点A收到客户端发送的命令后，计算key所属的槽id，若槽id属于节点A负责，则节点A会先尝试在自己数据库中查找键key，当发现此键key不存在，则会再检查槽id是否处于迁移状态，即clusterState.migrating_slot_to[槽id]是否为null，若clusterState.migrating_slot_to[槽id]不为空，则会向客户端返回一个ASK错误（ASK <槽id> <节点B的ip和端口>）
  3. 客户端接收到节点A返回的ASK错误后，解析并获取节点B对应的ip和端口，然后向节点B发送一个ASKING命令
  4. 节点B接收到ASKING命令后，后为此客户端设置一个REDIS_ASKING标识
  5. 客户端之后将之前发向节点A的命令再向节点B重新发一下
  6. 节点B接收到此命令后，会检查客户端的REDIS_ASKING标识是否设置，若已设置则会执行对应的命令并清除REDIS_ASKING标志位；否则会向客户端返回MOVE错误。

集群的复制

Redis集群中每个节点分为主机和从机，主机用于处理槽，从机用于复制主机。通过向节点B发送命令CLUSTER REPLICATE <节点A的node_id>，可以让节点B成为节点A的从机。具体的执行过程如下：

  1. 客户端向节点B发送CLUSTER REPLICATE <节点A的node_id>命令
  2. 节点B在接收到此命令后会将clusterState结构对象中mysel.slaveof指针属性指向记录A节点状态的clusterNode结构对象，以此来记录节点B为节点A的从机。slaveof的数据结构定义如下：

  2. 节点B之后会根据节点A中记录的ip端口和地址，向节点A发起复制命令进行复制操作

集群的故障检测

集群中每个节点都会定期向集群中其他节点发送PING消息以此来检测对方是否在线。如果节点A向节点B发送PING消息，由于网络等原因，节点B没有收到此PING消息或不能在规定时间内向节点A返回PONG消息，则节点A会将节点B标记为疑似下线状态（即节点A中标识节点B状态的clusterNode结构对象中flags属性REDIS_NODE_PFAIL标志位会被置1），同时节点A会将节点A认为节点B疑似下线报告通过gossip协议传播给集群中其他节点，如节点C。当节点C收到此消息后，会先找到节点C中记录节点B状态的clusterNode结构对象，然后在此结构对象中的fail_reports属性添加这条下线报告记录。具体数据结构定义如下：

集群的选主流程

我们知道，当某一主节点被判定为已下线状态，那么选主流程会保证从主节点对应的从节点中选举出新的主节点，选主流程的具体过程与哨兵机制中选主算法（Raft算法）相同，因此此处仅说明一下主要流程：

  1. 当从节点A发现自己复制的主节点被判定已下线状态时，会先将自己的选举纪元+1，然后从节点A会向集群广播一条CLUSTERMSG_TYPE_FAILOVER_AUTH_REQUEST消息，请求其他具有投票资格的主节点将选举票投给自己。
  2. 收到A节点CLUSTERMSG_TYPE_FAILOVER_AUTH_REQUEST消息的其他主节点，首先检查是否具有资格投票（即检查自己目前在集群中负责的槽数是否大于0），若具有投票资格且在当前选举纪元中未投过票，则会向节点A返回CLUSTERMSG_TYPE_FAILOVER_AUTH_ACK消息，表示同意节点A成为新的主节点；否则不响应或响应错误。
  3. 当从节点A接收到CLUSTERMSG_TYPE_FAILOVER_AUTH_ACK消息的数量大于N/2（N为集群中具有投票资格的主节点的数量），则表示此从节点被同意当选为新的主节点，之后此新的主节点将进行故障转移操作。
  4. 若在一个选举纪元内，所有从节点各自接收到CLUSTERMSG_TYPE_FAILOVER_AUTH_ACK消息的数量都不大于N/2+1，则表示此轮选举失败，会将选举纪元+1，重新开始新一轮的选举

集群的故障转移流程

当从节点被选举变为新的主节点后，后执行故障转移操作，故障转移流程如下：

  1. 被选中的从节点会执行SLOVEOF no one命令，将自己变为主节点
  2. 新的主节点会接管负责旧的主节点负责的槽，同时向集群广播一条PONG消息，让集群中其他节点知道旧的主节点被替换为新主节点，并且旧主节点负责的槽现在由新主节点负责
  3. 新的主节点开始接收和处理与自己负责的槽相关的命令请求

集群的消息机制

集群中各个节点通过发送和接收消息，来进行通信。消息包含消息头和消息体两部分，消息的数据结构定义如下（clusterMsgData.ping.gossip定义数组长度为1的详解请看数组长度为0或1原因）：

接收者可以根据发送者发送的这些信息，对接收者中记录的发送者状态进行更新。目前发送消息的类型主要以下五种：

  1. Gossip消息：MEET、PING、PONG消息采用Gossip协议（详情请看漫谈Gossip协议与其在Redis Cluster中的实现），每次发送MEET、PING、PONG消息时都会从自己已知节点列表中随机选取出两个节点，并将选出的两个节点的信息保存到clusterMsgData.ping.gossip属性中。MEET消息用于请求接收者将发送者加入到接受者所在的集群中，通常会在节点接收到客户端发送的CLUSTER MEET命令时，此节点会先解析出来命令的ip和port，然后向对应节点发起MEET命令。PING/PONG消息为集群中的心跳消息，用于集群间检测节点状态和传播节点信息。注意Gossip消息可以保证最终一致性，但可以大大减少集群间消息的数量。

  2. FAIL消息：当节点A发现主节点B被标记为已下线状态时，节点A会通过广播方式向集群广播一条主节点B的FAIL消息，从而可以尽快让集群里的所有节点知道节点B已下线。

  3. PUBLISH消息：当集群中的某个节点接收到来自客户端发送的PUBLISH 命令时，不仅会向对应channel频道发送message信息，同时还会向集群广播一条PUBLISH消息，使得所有收到这条PUBLISH消息的节点都向channel频道发送message消息。

    注意此处集群为了遵循集群中各个节点通过发送和接收消息来进行通信这一规则没有采用广播PUBLISH命令的方法，而主从节点复制PUBLISH命令时采用的是向所有节点广播PUBLISH命令。
