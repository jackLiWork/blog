G1垃圾回收

2020-08-27 java学习 G1垃圾回收, java 评论

G1算法简介

G1算法之前，CMS等垃圾回收算法都是将堆分为一个老年代和一个年轻代，这样每次都需要对年轻代或年老代整个区域都进行垃圾回收，这样就不能很好控制垃圾回收停顿时间。为了解决以上问题，G1算法采用将堆分为若干个区域，每个区域在在被使用时会标记为不同的类型（E表示Eden区域，S表示Survivor区域，O表示Old区域，H表示Humongous区域，其中E和S属于年轻代，O和H属于老年代，H为专门用于存放占用空间大于Region区域容量50%以上对象）。一个Region属于老年代还是新生代是动态的，在使用时候才会被标记，因此初始堆中的Region是没有标记的，并且在垃圾回收后完全被释放的Region也会被清除标记。当需要进行垃圾回收时，通过选取回收性价比高（垃圾占比高）的区域来到达控制垃圾回收停顿时间。

G1在标记过程中，使用三色标记法来避免重复扫描。开始前所有对象都为白色，当对象首次被扫描时将会被标记为灰色。
1、黑色：表示该对象为根对象，或者该对象的所有子对象都被扫描过（即该对象所有field都被标记完了）
2、灰色：对象本身处于扫描中，但该对象中的子对象没有被扫描完（即它的field还没有被标记完）
3、白色：未被扫描对象，当标记完成后，最终为白色的表示不可达对象，即垃圾对象

Java对象分配策略

我们知道，分配新对象在Eden区域中进行，当多个业务线程都去分配新对象时，不可避免存在竞争，因此为了减少多线程在分配对象时的竞争，java采用为每个线程分配一个私有的本地分配缓冲区（TLAB，位于Eden区，会参与Eden区的垃圾回收，默认为Eden空间的1%）的策略。当需要分配对象时，会先在当前线程的TLAB区域进行分配，由于TLAB区域为线程私有区域，因此只有当前线程可以在此区域内分配对象，但是注意其他线程可以读取此区域的对象数据。同时由于TLAB比较小，因此当TLAB的剩余空间不足以分配新对象时，会创建新的TLAB并进行空间分配或直接在Eden区的共享内存为此对象分配空间，此时分配的Region被标记为E；当Eden区也不足以分配对象时，会直接在老年代中分配空间，此时标记的对象被标记为O；注意当遇到大对象时，也会直接在老年代中分配空间，此时分配的Region被标记为H。Region的大小可以通过参数-XX:G1HeapRegionSize设定，取值范围从1M到32M，且是2的指数。每个 Region 实际又能被细分为最小为512字节的heap chunk，称为 card。每个card被它起始地址构所标记，成为card id，在整个 Heap中唯一，通过card id可以直接映射到对应card。

Young GC

新生对象都是在属于Eden的一个Region上进行分配，当此Region满了之后会从空闲region中再取一个新Region并标记为 Eden用于分配新生对象，直到标记为Eden的Region数达到上限，此时会触发young GC。young GC采用复制算法和可达性性分析法，在young GC整个处理期间都需要STW，支持并行处理。注意每次YGC都会遍历并回收全部被标记为Eden的Region，进行回收后会清除Eden标记，使这些Region重新变为无标记Region。同时在对Eden区域中对存活象进行复制操作时，也需要先获取一个新的Region并标记为Survivor后，再向此Survivor区域复制对象。

分析1：我们知道在处理老年代对象到年轻代对象的跨代引用时，如果从GC Roots开始扫描，那么不仅要扫描年轻代，还需要扫描整个老年代，会造成Young GC停顿时间过长。为此G1维护了一个从非收集空间指向收集空间的指针集合的RSet数据结构，对于Young GC来说，非收集空间指的是老年代，收集空间指的年轻代，因此RSet数据结构保存从老年代->年轻代的跨代指针。这样通过将每个年轻代Region对应的RSet作为root进行扫描就可以避免扫描整个老年代。GC Roots包括方法区中的静态属性引用和常量引用、虚拟机栈引用、本地方法栈引用。

分析2：我们知道不同Region被引用的数量是不同的，有的Region被引用的数量多，有的Region则被引用的数量少。当一个Region被引用数量特别多时，那么该Region对应的RSet中记录的引用数量就会特别多，造成占用内存较多。为了解决此问题，G1算法根据一个Region被引用的数量进行分级，不同分级记录采用不同粒度的记录方式，主要分为三级：sparse、fine和coarse。

  • 当RSet处在sparse级别，每个年轻代Region对应的RSet中，存储的为指向此Region的filed引用在老年代Region中所属card的card id集合。
  • 当RSet处在sparse级别，每个年轻代Region对应的RSet中，存储的为指向此区域的引用所在Region的Card位图集合，Card位图中每一位对应指向此区域的引用所在Region中一个card。
  • 当RSet处在sparse级别，每个年轻代Region对应的RSet中，存储的为指向此区域的引用所在Region的位图，此时需要通过扫描所有被引用的Region才能找出所有引用。

分析3：我们知道当代码执行赋值操作时，会改变Region间的引用关系，相关Region对应的Rset也需要改变，因此需要感知赋值语句来更新相关Rset。JVM通过在每个赋值语句后增加post-write barrier机制，来执行额外操作更新Rset，示意图如下：

由于赋值语句在应用线程中执行，如果将Rset的更新操作都放在应用线程中执行，就会影响应用系统的性能。JVM通过异步方式来处理，即将耗时的Rset更新操作放在GC refinement threads线程中去执行，应用线程与GC refinement threads线程间通过dirty card queue共享队列进行通信。在postwritebarrier函数中仅执行如下操作：

  1. 先判断filed指针本身是否来自老年代和new_value指向对象是否来自年轻代

  2. 如果filed指针本身来自老年代且new_value指向的对象来自年轻代，则将filed指针所在card和field指针指向的空间所在Region等信息加入dirty card queue中

GC refinement threads线程会不断去不断从dirty card queue共享队列中顺序解析出对应信息并完成RSet更新工作，包括RSet粒度升级等工作。GC refinement threads是持续运行的，并且线程数会随着dirty card queue的队列长度进行动态调节。由于是异步消费，那么就会存在GC refinement threads消费速度小于dirty card queue增长速度，dirty card queue会出现积压，此时应用线程在往dirty card queue中增加新的dirty card时，也会从dirty card queue中消费一条dirty card，此时应用的性能必定会受到影响，因此要避免这种状况发生。

Mixed GC

当整个堆内存（包括老年代和新生代）使用率超过设置IHOP值（默认是45%，通过-XX:InitiatingHeapOccupancyPercent进行设置）后，Mixed GC回收过程会被启动。具体检测堆内存使用情况的时机是年轻代回收之后或者houmongous对象分配之后。Mixed GC采用增量式垃圾回收策略，处理过程可以分为两个阶段：全局并发标记阶段和混合收集阶段，注意一个并发标记后可以有多个清理阶段。由于Mixed GC采用复制算法，因此解决了CMS中存在的内存碎片问题。

全局并发标记阶段（global concurrent marking）

分析1：我们知道不管是Young GC还是Mixed GC在通过可达性分析策略标记存活对象时，都需要STW并扫描GC Roots，且扫描的GC Roots相同。如果Mixed GC能够在Young GC通过GC Roots扫描其直接引用对象并判断分代时，也将属于老年代的直接引用进行记录，Mixed GC就可以少一次STW，也就是Mixed GC的初始标记阶段。因此当整个堆内存使用率达到IHOP值时，Mixed GC并不会立即发起，而是等待下一次年轻代收集，利用年轻代收集的STW时间段，完成初始标记，这种方式称为借道(Piggybacking)。

分析2：我们知道Mixed GC在并发标记过程中，应用程序会与GC线程并发工作，而应用程序会改变已标记对象间的引用关系，若不进行额外标记，则会造成漏标或错标，错标会产生浮动垃圾，漏标则会导致在随后的回收阶段使正在使用的对象被回收，影响应用程序的正确运行。为了解决此问题，Mixed GC采用SATB write barrier中pre-write barrier技术来解决漏标问题。同时为了尽量减少pre-write barrier操作对业务线程性能的影响，业务线程仅将涉及到的赋值前的值信息（log）添加到自己的SATBMarkQueue队列里，当SATBMarkQueue队列满了之后，SATBMarkQueue队列就会被加到全局的SATB队列集合SATBMarkQueueSet中，同时给当前业务线程分配一个新的SATBMarkQueue队列。并发标记线程会定期检查SATBMarkQueueSet集合的大小。当SATBMarkQueueSet集合中数量超过一定阈值后，并发标记线程就会处理集合里的所有队列，即并发标记线程将队列里记录的每个oop为根，通过递归遍历标记。

white对象在并发标记阶段会被漏标的充分必要条件（满足其中一个即可）为：
1、业务线程插入了一个从black对象到white对象的引用，如红色实线所示
2、业务线程删除了所有从grey对象到white对象的直接或者间接引用，如红色虚线所示

防止漏标的方法就是破坏上述充分必要条件：

1、CMS采用利用post write barrier将所有新插入的引用关系都记录下来，从而破坏了第一个充分必要条件
2、G1采用利用pre write barrier所有即将被删除的引用关系的旧引用记录下来，从而破坏了第二个充分必要条件

分析3：我们知道业务线程在运行过程中会不断产生新的对象，但pre write barrier并不能处理分配新对象的情况，此部分对象也需要被标记。为了解决这个问题，G1采用快照的方式来保证Mixed GC开始后新增的对象不会被回收。同时由于Mixed GC支持增量式标记，因此需要维持两个指针prevTAMS和nextTAMS，在每次Mixed GC初始标记的结束阶段（此时会STW），会更新nextTAMS为top，即[prevTAMS, nextTAMS)为本轮需要标记的区域；在Mixed GC的并发标记结束阶段，会更新prevTAMS为nextTAMS，表示当前轮的并发标记结束。如下图：

  • [bottom, prevTAMS): 此部分在n-1轮被标记，对象存活信息存储在prevBitmap中
  • [prevTAMS, nextTAMS): 此部分在第n轮被标记，对象存活信息存储在nextBitmap中，直到Mixed GC的并发标记结束才可以使用，此时nextBitmap也就变为prevBitmap
  • [nextTAMS, top): 此部分对象表示在第n轮进行并发标记过程中产生的新对象，因此在第n轮垃圾回收过程中是隐式存活的
  • [top, end): 此部分表示当前Region未使用的部分

结论：分析中使用的pre write barrier技术和快照技术，合称为G1算法的SATB，SATB保证了G1算法正确性，但是会产生浮动垃圾：

  1. 所有在 concurrent marking 阶段开始时活的对象，一定会被标记和递归遍历对象中引用（注意开始时存活的对象后来变为垃圾，也会被标记）；
  2. 所有在 concurrent marking 过程中产生的新对象一定被标记（注意新对象后来变为垃圾，也会被标记）;

并发标记的具体步骤：

(1) Initial Mark初始标记 STW

首先暂停所有业务线程，即STW；接着标记GC ROOTS直接可达的对象，并将它们的属性字段压入扫描栈中等待后续扫描。G1使用外部的bitmap来记录标记信息。由分析1可知此阶段会借助YGC的STW实现，但是与YGC在逻辑上相互独立。

(2)Root Region Scanning 根区域扫描 并发扫描

根区域扫描是从Survior区的对象出发，标记此对象属性中引用的老年代对象，并把这些老年代对象压入扫描栈中等待后续扫描。由于此阶段从Survior区开始标记，因此Root Region Scanning必须在下一次YGC开始前完成。

(3)Concurrent Marking 并发标记

不断从扫描栈取出引用，并递归扫描引用对象里的属性，知道直到扫描栈为空。过程中还会递归扫描SATB write barrier所记录下的引用。注意Concurrent Marking 可以被YGC中断

(4)Remark 最终标记 STW

STW操作。在完成并发标记后，每个Java线程还会有一些剩下的SATB write barrier记录的引用尚未处理。这个阶段就负责把剩下的引用处理完。同时这个阶段也会对弱引用进行处理。注意这个暂停与CMS的remark有一个本质上的区别，那就是这个暂停只需要扫描SATBMarkQueueSet中dirty card，而CMS的remark需要重新扫描dirty card以及整个根集合（包括所有年轻代对象），因而CMS remark有可能会非常慢。注意G1 Mixed GC的最终标记和CMS的重新扫描都是采用三色标记法，即遇到黑色对象就停止递归扫描，因此其他阶段对存活对象的标记越完整，最终标记和重标记阶段花费的时间越短。

(5)清理阶段 需要STW

经过标记之后，collector就知道哪些Region有存活的对象，首先清点出有存活对象的Region和没有存活对象的Region，并更新RSet。对于有存活对象的Region，G1会根据统计预测模型找性价比高的若干Region，组成的集合就叫做CSet。对于那些完全可回收的Region(没有存活对象)，则收集起来加入到可分配Region队列实现直接回收。在Mixed GC中的CSet是选定所有年轻代中的region，外加根据统计预测模型得出满足暂停时间的若干年老代region。（在YGC中的Cset是选定所有年轻代中的region，通过控制young gen的region个数来控制young GC的开销）

混合收集阶段：

为了满足停顿预测模型即暂停时间，G1可能会产生连续多次的混合收集（默认8次，通过-XX:G1MixedGCCountTarget设置），并与应用线程交替执行回收，G1会优先回收垃圾多的内存区域。对于要回收的分区，会将该分区中存活的数据拷贝到另一个分区。在一次混合收集中可以回收的分区数取决于三个因素：（1）有多少个分区被认定为垃圾分区，配置参数
-XX:G1MixedGCLiveThresholdPercent=n表示如果一个分区中的存活对象比例超过n，就不会被挑选为垃圾分区（2）可以通过-XX:G1MixedGCCountTarget=n参数（默认是8）设置在一个并发周期中最多经历几次混合收集周期，如果减小这个值，可以增加每次混合收集收集的分区数，但是可能会导致停顿时间过长；（3）通过MaxGCPauseMillis`参数可以设置期望的GC停顿的时间，默认值是200ms，在混合收集周期内的停顿时间是向上规整的，如果实际运行时间比这个参数小，那么G1就能收集更多的分区。

转移失败

当G1垃圾回收发生转移失败时，便会执行Serial GC（全程STW），转移失败的条件如下：

  • 从Eden区转移存活对象到Survivor区或从Survivor区向Old区晋升对象时，无法在老年代找到可用的空闲分区
  • 从Old区复制存活对象到空闲Old区时，无法找到可用的空闲Old区
  • 分配巨型对象时在H区无法找到足够的连续空间

避免Evacuation Failure，考虑以下选项。

  • 增大heap堆大小

  • 如果从Eden区转移存活对象到Survivor区有问题，可以增大-XX:G1ReservePercent=n参数，默认是10，G1会预留为Survivor预留更多空间

  • 增大并行标记的线程数，使用-XX:ConcGCThreads=n选项，减少并发标记时间以更快进入清理阶段

G1的最佳实践

  1. 关键参数项
  • -XX:+UseG1GC，告诉JVM使用G1垃圾收集器
  • -XX:MaxGCPauseMillis=200，设置GC暂停时间的目标最大值，JVM会尽力达到这个目标
  • -XX:INitiatingHeapOccupancyPercent=45，设置触发Mixed GC时整个堆的使用率
  1. 最佳实践
  • 不要设置年轻代和老年代的大小，通过-Xmn显式设置年轻代的大小，会干扰G1收集器的默认行为：
    • G1不再以设定的暂停时间为目标
    • G1不能按需扩大或缩小年轻代的大小
