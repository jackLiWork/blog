Kafka事务

2020-08-20 Kafka学习 Kafka, Kafka事务 评论

应用场景

最初出发点是为了在Kafka Streams中实现Exactly-Once语义的数据处理，后来发展为允许将消费和生产的batch处理（涉及多个Partition）在一个原子单元内完成（即操作要么全部完成，要么全部失败）

事务核心变量

ProducerId：生产者id，由事务协调器提供，生产者重启后会发生改变，因此通过ProducerId和主题-分区序列号sequence number（ 从0开始自增，为每个 Topic-Partition都会维护一个独立的 sequence number）仅能保证会话级别的幂等性（即保证单个会话且单个主题-分区内不丟不重）。为了保证此PID与sequence number对应信息在重启后快速灰度，broker增加了.snapshot的快照文件，使得broker在重启后可以加载最新.snapshot文件进行恢复到某一时间点，在通过读取少量增量log日志信息实现完整恢复。

TransactionalID：事务id，由生产者应用自己提供，重启后不会改变，主要作用是从事务协调器角度可以跨会话唯一标记一个生产者（即事务协调器会维护事务id与生产者id的映射关系，当生产者重启后或此生产者宕机引入新的生产者时，分配的生产者id不变）

ProducerEpoch：生产者纪元，每次生产者去事务协调器获取生产者id时，此值都会加1，主要用于保证每个事务id在同一时间仅有一个生产者实例（旧的生产者会抛异常退出）

消息中Transactional字段：事务消息数据协议中会新增一个字段用于告诉broker此条消息是一个事务消息，数据协议格式如下：

Transaction Marker消息：用于事务提交的最后阶段，由事务协调器向broker发送，表示当前事务被提交或回滚，相当于一个信号，这样在Consumer处理消费时，就可以根据 marker信息来识别marker消息之前的事务消息是被提交还是被回滚

Last Stable Offset (LSO)：对于一个 Partition 而言，offset 小于 LSO 的数据，全都是已经确定的数据，主要应用于事务操作，即offset 小于LSO的事务操作都是已经完成的事务（已经 commit 或 abort）。如果此Partition没有涉及到事务消息，那么 LSO 就是HW（表示已经成功备份到其他 replicas中且满足最小备份数量的最新一条数据的offset）。如果consumer读取消息且设置隔离界别为读已提交，则此consumer仅能看到offset小于LSO的消息。

事务协调器

事务协调器通过2PC一致性协议实现事务管理，同时维护了事务id与生产者id映射关系以及每个生产者当前处理事务的状态和状态转移。为了保证高可用，复用kafka中主题的多副本和acks参数机制，即通过内部名为__transaction_state的主题来保存每个事务的状态（默认为50个分区，通过事务id的hash值取余来获取对应分区），主题中消息存储的格式如下：

key 含义
TransactionalId 生产者的事务id，由生产者提供，重启后不会发生变化
value 含义
ProducerId 生产者id（Pid），用于标识一个生产者（全局唯一），由事务协调者提供。若事务id对应的生产者id已存在，在会返回此事务id绑定的生产者id，但ProducerEpoch会递增
ProducerEpoch 生产者id对应的当前纪元，每次获取Pid时ProducerEpoch都会进行递增，由事务协调者提供
TxnTimeoutDuration 事务的超时时间
TxnStatus 事务的状态
TxnPartitions 当前事务涉及改动的主题和对应主题的分区组成的列表
TxnLastUpdateTime 事务的更新时间
TxnStartTime 事务的开始时间

从图中可以看到，主要分为两条线：事务正常提交（蓝色线）和事务异常回滚（红色线），同时我们也可以看到只有Empty->OnGoing、OnGoing->PrepareCommit、OnGoing->PrepareAbort、OnGoing->PrepareEpochFenced状态转换需要与客户端交互，其他的状态转换都是在事务协调器与broker之间交互完成的。这也说明，事务状态一旦转移到PrepareEpochFencd、PrepareAbort和PrepareCommit状态，事务的结果已经确定，不需要客户端参与（客户端挂了也没有关系）。同时我们也可以得出当事务协调器记录的事务状态处于OnGoing状态时，此时若客户端由于某种原因挂掉了，那么客户端在重启之后，由于ProducerEpoch会递增加1，导致原来的事务状态转移到PrepareEpochFencd状态，接下来此事物会被强行停止。

事务的核心流程

1、查找事务协调器TransactionCoordinator

生产者通过向集群中负载最小的节点Broker发送FindCoordinatorRequest请求以获取对应的事务协调器TransactionCoordinator。对于具有事务Id的生产者，通过hash(transactionalId)/50来获取对应主题__transaction_state的分区，从而获取对应事务协调器所在的broker服务器

2、获取生产者id

生产者在initializeTransactions()方法中会向 TransactionCoordinator发送 InitPidRequest请求以获取其分配的生产者id（PID），并持久化对应的TransactionMetadata到事务日志中（主要保存事务id与PID的映射关系）
（1）如果此生产者拥有的事务id在当前事务协调器中不存在对应的事务状态，那么会产生一个新的生产者Id且初始epoch为-1，同时初始化事务状态对象TransactionMetadata，并持久化到事务日志中
（2）如果此生产者拥有的事务id在当前事务协调器中已存在对应的事务状态，会进行状态检查，检查通过会将TransactionMetadata持久化到事务日志中

当前状态 执行操作
PrepareAbort 或 PrepareCommit 返回并发事务异常
CompleteAbort或CompleteCommit 转移为Empty状态，并递增ProducerEpoch值，并将状态持久化到事务日志中
Ongoing 转移为PrepareEpochFence状态，并回滚当前事务，最后返回并发事务异常
PrepareEpochFence或Dead 直接抛出FATAL异常

3、开始事务

生产者通过调用 beginTransaction()开始一个事务操作，注意此处并没有与事务协调器进行交互，只是更新了本地事务状态

4、处理事务

生产者会进行具体的事务处理，包括发送消息或消费消息。

（1）作为生产者，在调用send()方法，会收集事务处理过程中涉及的主题-分区信息，并通过AddPartitionsToTxnRequest请求将这些主题-分区信息发送到事务协调者TransactionCoordinator，事务协调者收到请求后会将这些主题-分区信息先更新到TransactionMetadata对象中，然后在持久化到事务日志中。
（2）作为消费者，在调用sendOffsetsToTransaction()方法时，会收集事务处理过程中涉及的主题-分区信息，通过AddPartitionsToTxnRequest请求将这些主题-分区信息发送到事务协调者TransactionCoordinator，事务协调者收到请求后会将这些主题-分区信息先更新到TransactionMetadata对象中，然后在持久化到事务日志中；当消费者收到事务协调器TransactionCoordinator的响应后，会再次发送TxnOffsetsCommitRequest请求给消费者所在的组协调器GroupCoordinator，组协调器GroupCoordinator 在收到AddPartitionsToTxnRequest请求响应后，会将 offset 信息持久化到consumer offsets log中（包含对应的PID信息），但是不会更新到缓存中（保证这个 offset 信息对 consumer 是不可见的）。

5、提交事务

当生产者需要提交事务或回滚事务时，生产者会向事务协调器TransactionCoordinator发送EndTxnRequest请求，TransactionCoordinator 在收到 EndTxnRequest 请求后，将状态转移成 PREPARE_COMMIT 或 PREPARE_ABORT，更新事务的TransactionMetadata对象信息，并将TransactionMetadata对象信息持久化到事务日志中，之后事务协调器会进行以下操作（不涉及客户端）：

（1）事务协调器向TransactionMetadata对象信息保存主题-分区所在的Broker发送WriteTxnMarkerRquest请求。若是作为生产者时涉及的Topic-Partition，则会在对应主题分区中插入一个Transaction Marker消息；若是作为消费者时涉及的Topic-Partition（即主题__consumer_offsets），则也会向此主题分区中插入一个Transaction Marker控制消息；
（2）当这个事务涉及到所有 Topic-Partition都已经进行完对应操作后，TransactionCoordinator 会将这个事务的状态置为 COMMIT 或 ABORT，并持久化到事务日志文件中

6、consumer消费事务消息

对于事务，consumer端消费者隔离级别有两种：读已提交read_committed和读未提交read_uncommitted。（1）对于read_uncommitted隔离级别，consumer能够读取到所有数据，与普通的消费机制基本相同，即会返回HW之前的所有消息（2）对于read_committed隔离级别，当消费者向broker发送Fetch请求拉取消息数据时，server端只会返回LSO之前的数据。同时我们有事务的提交机制可知，事务数据中不仅包含生产者提交的消息，还包括Transaction Marker控制消息和回滚的消息，因此消费者需要进行check来过滤掉Transaction Marker控制消息和回滚的事务消息。为了提升消费者的过滤Transaction Marker控制消息和回滚的消息性能，kafka的broker端会追踪每个Partition涉及到的回滚transactions的相关信息，并将其持久化到一个单独的文件中（文件后缀为.txnindex），存储的数据如下：

变量名 含义 变量名 含义
PID 生产者id LastOffset 此回滚事务涉及消息中的最大偏移量
FirstOffset 此回滚事务涉及消息中的最小偏移量 LastStableOffset 此回滚事物在提交时对应的LSO
