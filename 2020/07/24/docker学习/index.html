docker介绍

Docker 是一个开源的应用容器引擎，让开发者可以打包他们的应用以及依赖包到一个可移植的容器中,然后发布到任何流行的Linux机器或Windows 机器上。容器是完全使用沙箱机制，相互之间不会有任何接口。常见的应用场景有：使应用的打包与部署自动化、创建轻量且私密的PAAS环境、实现自动化测试和持续的集成/部署等。如下为docker容器典型的应用场景：

docker特性

容器本身轻量化，占用宿主机资源较少

单台机器可以运行多个docker

部署在不同docker上的机器保证相互隔离

通过构建docker镜像保证开发与线上环境一致

docker底层原理

docker是通过使用linux底层提供的namespace和control groups机制进行实现的，下面将对这个这两个机制进行讲解，来说明docker的实现原理。

1、Linux中的namespace

Linux通过提供Namespace机制，实现PID、IPC、Network等系统资源的局部性和隔离性，即属于不同的Namespace的系统资源相互不可见。在操作系统层面，我们可以看到，会存在相同的进程id，但是它们肯定属于不同的namespace。当在某一namespace下时，执行linux命令如ps，我们只能看到此namespace下的相关资源信息，使得每个namespace看上去就像是一个单独的Linux系统。

Mount namespaces：Mount namespace用来隔离文件系统的挂载点, 使得不同的mount namespace拥有自己独立的挂载点信息，不同的namespace之间不会相互影响，有利于构建用户或者容器自己的文件系统目录

UTS namespaces：提供了主机名和域名的隔离，能够使得子进程有独立的主机名和域名，使得docker容器可以被视为集群中一个独立的节点

IPC namespaces：IPC是Linux下进程间通信的方式，有共享内存、信号量、消息队列等机制。通过IPC namespaces我们实现进程只能与同一namespaces下进行IPC通信。

PID namespaces：用于隔离进程的PID，使得不同namespace里的进程ID可以重复且相互之间不影响。

Network namespaces：用于隔离网络相关的系统资源，使得每个容器有其独立的网络设备、IP 地址、IP 路由表、端口号等，从而实现在同一服务器上部署在不同容器中的应用可以绑定到相同的ip地址和端口上。

User namespace：用于隔离用户和用户权限，同一用户在不同的容器可以具有不同的角色和权限。

2、linux中control group

linux中control group用于对一组进程进行系统资源的分配和控制，系统资源包括：cpu核数、cpu占用时间、内存使用大小、网络带宽、文件系统缓存等。通过control group，我们在同一服务器上为不同docker容器定制化不同的系统资源，从而实现资源的合理分配和高效利用，如对cpu密集型分配较多的cpu资源、对于网络I/O密集型分配较多的网络带宽资源。更典型的应用是，对于云计算来说，可以通过容器化实现服务器资源的定制化分配和收费。

3、Union File System

由于docker中镜像是分层的，因此需要一种技术，将多个不同位置的目录挂载到同一目录下，让用户感觉像是在真正的用户空间roots下。UnionFS，就是是将多个不同位置的目录联合挂载（union mount）到同一个目录下的技术。深入了解请看UFC和AuFs在Docker中应用

docker镜像

linux操作系统分为内核空间（kernel）和用户空间（rootfs，包含我们熟悉的 /dev, /proc, /bin 等目录）。而Docker镜像，就相当于Docker容器的rootfs，每个Docker容器都有自己的镜像，即用户空间，但是公用宿主机的内核空间，示意图如下：

从图中可以看到，Docker镜像包含镜像层和容器层，镜像层除了提供容器运行时所需的程序、库、资源、配置等静态文件外，还包含运行时的一些静态配置参数（如匿名卷、环境变量、用户等），镜像层是只读层，其内容在构建之后就不会被改变。镜像层采用分层结构，上层的镜像层需要依赖下层镜像层，因此越下层的镜像层其功能越基础。由于镜像层是只读的，如果有多个容器依赖了相同的镜像层，就可以实现镜像层在不同容器间的共享，这样就可以保证只在内存和硬盘中加载一份就可以。

容器层只有在镜像被启动的时候，才会在镜像顶层生成。容器中所有对文件的改动都发生这一层，其具体操作流程如下：

  • 容器读取文件时：

    先在容器层中进行查找，如果找到直接使用容器层中的文件。否则，从镜像层最上层开始往下找，若已经在内存中，可以直接使用，否则找到后加载到内存中，这样可以保证运行在同一台机器上的Docker容器共享相同的镜像文件

  • 容器修改文件时：
    若文件在容器层，则可以直接修改。否则先从镜像层找到对应文件，然后复制到容器层，最后直接修改容器层的文件。

  • 容器删除文件时：
    若文件在容器层，则删除容器层的文件，并在容器层标记此文件为删除状态，若不在容器层，直接在容器层标记此文件为删除状态。所以删除为软删除，在镜像层中对应文件仍存在，但对容器层不可见。

  • 容器增加文件时：
    直接在容器层进行增加即可

docker镜像构建

docker镜像构建时，会一层层构建，前一层是后一层的基础。每一层构建完就不会再发生改变，后一层上的任何改变只发生在自己这一层。比如，删除前一层文件的操作，实际不是真的删除前一层的文件，而是仅在当前层标记为该文件已删除。在最终容器运行的时候，虽然不会看到这个文件，但是实际上该文件会一直跟随镜像。因此，在构建镜像的时候，需要额外小心，每一层尽量只包含该层需要添加的东西，任何额外的东西应该在该层构建结束前清理掉。
