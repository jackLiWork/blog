数据库扩展性设计与实践

2020-07-15 mysql学习 分库分表, 垂直拆分, 数据库扩展性设计 评论

垂直拆库

按照业务模式或业务领域（如用户领域、商品领域、订单领域等），将不同业务领域的表划分到不同的数据库中，同一业务领域的表划分到同一数据库中
优点：

  1. 拆分后实现各业务系统间解耦，业务独立维护，各业务之间互不干扰
  2. 各业务易于水平扩展
  3. 高并发情况下，能一定程度上突破IO、连接数、单机硬件资源的瓶颈。

缺点：

  1. 部分业务表无法join，只能在应用层通过编程方式解决，提高了系统复杂度
  2. 受每种业务不同的限制存在单库性能瓶颈
  3. 涉及多个业务，需要使用分布式事物

垂直拆表

将具有较多字段的表拆分为多个具有较少字段的表，通常拆分后会分一个主表和多个分表，各分表之间通过主表的主键进行关联

拆分依据：

（1）将长度较短，访问频率较高的字段尽量放在一个表里，这个表暂且称为主表（数据库缓存是以row为单位的，短row可以增加缓存中数据量）

（2）将字段较长，访问频率较低的字段尽量放在一个表里，这个表暂且称为扩展表

（3）经常一起访问的属性，可以放在一个表里（提升缓存命中率）

（4）如果字段确实过多，主表和扩展表都可以有多个

优点：

  1. 可以使得每条记录数据变小，一个数据块（block）就能存放更多的数据，在查询时会减少I/O次数

  2. 可以达到最大化利用Cache的目的，拆分时可以将不常变的字段放在一个表，将经常改变的字段放一个表

  3. 数据维护简单

缺点：

  1. 由于多个分表需要主键关联，主键会出现冗余，需要管理冗余项
  2. 查询涉及多表数据时，尽量在应用层进行join(增加CPU开销，不要使用数据库JOIN操作)
  3. 存在单表数据量过大的问题
  4. 插入涉及多表记录或强相关数据时，需要使用事物
  5. 拆分工作量较大

服务层尽量不要使用mysql的Join关键字的原因：

（1）join更消损耗数据库性能

（2）join会让关联的表耦合在一起（必须在一个数据库实例上），不利于库表的后续扩展

水平分库分表

将同一个表根据某些策略将表中的记录拆分到不同的库和表中，每个库表中包含一部分数据。

优点：

  1. 应用端改造较少（需要中间件支持）
  2. 在高并发和海量数据的场景下，能有效缓解单机和单库的性能限制带来的压力，突破IO、连接数、硬件资源的瓶颈

缺点：

  1. 涉及多个库的数据需要采用分布式事务
  2. limit分页、排序、跨库join等处理复杂
  3. 生成主键Id时需要保证全局唯一，实现复杂
  4. 需要处理路由问题

拆分策略：

  1. 范围法，适用于OLAP场景，如数据仓库、日志数据、hive表、交易流水时间等
    • 扩容简单
    • 拆分的列需要保证递增型（如添加时间或更新时间）
    • 拆分后，每个库表中的数据量分布不均
    • 拆分后，每个库表中数据请求量分布不均（通常新数据会有更高的请求量）
  2. Hash分片，适用于OLTP场景，大多数以主查询纬度的ID主键进行拆分（通常通过userId、orderId进行分库分表）
    • 拆分后，每个库表中的数据量分布较均匀
    • 拆分后，每个库表中数据请求量分布较均匀
    • 扩容复杂，扩容时会进行重新hash，需要进行数据迁移

拆分依据：

  1. 单表容量超过1000w行，需要进行分表，分表数=（未来3到5年内总共的记录行数) / 单张表建议记录行数 （单张表建议记录行数 = 1000万）
  2. 单库容量超过300G，需要进行分库，分库数量 = （3到5年内的存储容量）/ 单个库建议存储容量

分库分表sql执行步骤（假设库个数为m，表个数为n）：

  • sql解析：首先对sql进行解析，得到需要插入记录的id字段值num
  • sql路由：sql路由包括库路由和表路由。库路由用于确定这条记录应该插入哪个库（num%m），表路由用于确定这条记录应该插入哪个表(num/m%n)。
  • sql改写：由于不同的记录可能会被插入到不同的库表中，因此需要对sql进行改写，确保插入记录位置的正确性。
  • sql执行：一条sql经过改写后变成了多条sql，为了提升效率应该并发的到不同的库上去执行，而不是按照顺序逐一执行
  • 结果集合并：每个sql执行之后，都会有一个执行结果，我们需要对分库分表的结果集进行合并，从而得到一个完整的结果。

跨库排序的解决方案：

  1. 暴力法：如查出排序的n条数据，则先每个库查出n条数据，然后在内存中排序
  2. 禁止跳页查询：只提供下一页的功能，每次记录当前页最后一条记录的主键ID
  3. 允许数据精度损失：如查出排序的n条数据，每个库各取n/2条数据，并在内存中进行归并排序

单表数据迁移多库多表步骤：

  1. 通过dump方式，将存量数据导入到分表中，并上报此时binlog的位置，并同时开始记录增量数据
  2. 存量数据同步完以后，将记录的增量数据同步到分表中
  3. 当增量数据同步追上最新数据后，同时对原表和分表写入
  4. 停止增量数据同步，依靠双写保证一致性
  5. 业务修改代码切换到分表上，双写继续保留一段时间
  6. 分库分表功能验证无误后，全量切换到分表上

读写分离

采用一主多从实现读写分离，主库用于承接写流量和部分读流量，通过主从同步机制实现从库的同步，从库仅承接读对量

优点：

  • 线性提升数据库读性能
  • 通过从库承接主库的大部分读流量，提升主库的写性能

缺点：

  • 在读取数据时，需要考虑主从延时
  • 存在潜在的数据一致性问题

使用场景：

  • 读访问量很大；
  • 高读写比（同一时间有大量的读操作和较少的写操作）；
  • 读操作对数据的实时性，一致性要求没有那么高。

数据库总体拆分原则

  1. 优先考虑缓存降低对数据库的读操作
  2. 再考虑读写分离，降低数据库写操作
  3. 再考虑按照业务垂直拆分
  4. 再考虑水平拆分：先分库(设置数据路由规则，把数据分配到不同的库中)
  5. 最后再考虑分表，单表拆分到数据1000万以内。
