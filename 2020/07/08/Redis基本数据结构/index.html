字符串

Redis中字符串采用Struct方式实现，又称为简单动态字符串(simple dynamic string, SDS)，是Redis中最基础的类型，悲被广泛使用，如redis中所有键、AOF缓冲区、客户端状态的输入缓冲区等。SDS通过增加free属性并采用空间预分配和惰性释放功能来提高性能，减少内存重分配次数。通过len判断是否结束而不是采用空字符来保证二进制安全，防止内存泄漏。struct定义如下：

SDS的空间分配策略：

  • 当需要对SDS修改且发现空间不够用时，如果需要的空间长度（len属性）小于1M，那么也将额外分配同样大小的未使用空间，即free=len，空间的总大小为 （2*len + 1）

  • 当需要对SDS修改且发现空间不够用时，如果需要的空间长度（len属性）大于等于1M，那么也将额外分配1M的未使用空间，即free=1M，空间的总大小为（len + 1M）

  • 当需要缩短SDS字符串时，仅仅通过增加free属性值将多余的使用空间变为未使用空间，但是不会进行内存释放操作

字典数据结构

可以看到Redis字典结构中核心数据结构为dictht，dictht的实现方式是链地址法Hash实现，dictht结构体的定义如下：

hash值计算方式：Redis使用的hash计算方法为MurmurHash算法，它的主要优点在于高性能算法，低碰撞率。

rehash原因：当哈希表由于容量不够用或者容量空闲较多时，需要进行对应的扩容或者缩容操作，由于扩容或缩容操作会造成hash表中掩码属性（sizemask）的变化，使得容量变更前后相同的key对应value被映射到不同的表索引中，因此需对所有键值重新计算索引值并更改值映射位置，此过程被称为rehash。判断是否进行扩容和缩容操作的阈值，称为负载因子，即键值对数量（dictht.used）/哈希表容量(dictht.size)。

rehash的条件：

  • 当服务器没有执行bgSave或bgRewriteof命令时，且负载因子 > 1时，会进行rehash扩容操作

  • 当服务器正在执行bgSave或bgRewriteof时，且负载因子 > 5时，会进行rehash扩容操作（执行bgSave或bgRewriteof命令时，服务器会创建子进程，由于写时复制机制（注意父子进程更改都会触发）的存在，此时进行扩容操作会触发内存复制，因而可以通过提高负载因子来减少不必要的内存复制操作）

  • 当负载因子 < 0.1时，会进行rehash缩容操作

为了避免rehash的过程对服务器性能的影响，Redis采用渐进式扩容方式。渐进式rehash具体过程如下：

  1. 为ht[1]分配空间，当执行扩容操作时，ht[1]的容量大小为第一个大于等于ht[0].used*2的2^n；当执行缩容操作时，ht[0]的容量大小为第一个大于等于ht[0].used的2^n
  2. 维持一个索引计数器变量rehashidx，初始将它的值设置为0，表示rehash 开始，每次rehash完当前bucket（即ht.table[rehashidx]）上所有键值对后，将rehashidx+1，然后继续处理下一个bucket
  3. 由于bucket是一个链表式结构，所以需要循环遍历bucket上的所有元素进行rehash操作。在对单个键值对进行rehash操作时，需要重新计算此key在ht[1]中索引位置i（需要根据ht[1]的掩码进行按位与计算），然后将对应的key-value设置到ht[1]中对应的bucket中（即ht[1].table[i]），然后删除ht[0]中对应的key-value对，更新ht[0].used减1，更新ht[1].used加1。处理完一个bucket后，将ht[0].table[dict.rehashidx] 置为 NULL
  4. 在rehash期间，当对字典执行新增操作时，仅在hash表ht[1]上执行，保证ht[0]的数据只减不增；当对字典执行删除、更新、查找操作时，会先在hash表ht[0]计算出对应key所在的bucket索引位置i，然后将i与rehashidx进行比较，如果i<rehashidx，说明元素已经迁移到ht[1]中，将在ht[1]上进行相应操作；当i>=rehashidx，说明元素未迁移到ht[1]中，将在ht[0]上进行相应操作，注意此时由于rehash期间的新增元素是在ht[1]上进行，存在ht[0]上找不到对应对应元素，故需要先在ht[0]里面找一遍，如果找不到再到h[1]里面找，以此判断需要在哪个hash表上进行操作。
  5. 当ht[0]中所有键值对都迁移到h[1]中时，将h[0]中table指针指向的空间释放，然后赋值为h[1]，最后新建一个空白哈希白表，并赋值为h[1]
  6. rehash结束后，将rehashidx 设置成-1

渐进式哈希的控制：

  • 控制每次rehash的bucket个数，支持设置，默认为1，如果bucket为null，则继续处理下一个，但是跳过为null的bucket个数不能超过10*n
  • 控制每次rehash的时长，支持设置超时时间，在时间段内每次rehash的bucket个数为100，连续为null的1000个bucket算为一个bucket。如果超时还未处理完，将结束rehash

整数集合

整数集合（intset）是Redis用于保存整数值集合的数据结构，查找元素复杂度O(logN)，它可以存储的类型有int16_t、int32_t、int64_t。当一个集合中只包含整数，且这个集合中的元素数量不多且无重复时(默认不超过512)，redis就会使用整数集合intset作为集合的底层实现。其定义的struct结构如下：

需要注意的是，如果插入数据的编码类型比当前intset的编码类型要大，为了能存储新元素，intset需要进行类型扩展。扩展类型时会把intset中原有的所有元素都扩展成新的类型，这样就需要重组intset。同时插入或删除元素时，可能需要移动元素。因为当元素数量较多时，移动或重组比较耗时，因此intset不适合存储大量元素

压缩列表

压缩列表是Redis为了节约内存而设计的线性数据结构，每个元素可以是一个字节数组或者整数值，哈希、有序集合中都有用到压缩列表，当一个哈希或有序集合中元素数据少且都是短字符时则会用压缩列表作为底层实现（会产生连锁更新，平均时间复杂度为O(N)）。ziplist的struct结构定义如下：

字段名 占用字节数 说明
zlbytes 4 压缩列表的字节长度，压缩列表最长(2^32)-1字节；
zltail 4 压缩列表尾元素相对于压缩列表起始地址的偏移量
zllen 2 压缩列表中元素数目，当超过最大长度时会遍历整个压缩列表来获取元素数目
previous_entry_length 1、5 表示前一个entry元素占用的字节数，用5字节表示时（首字节为固定为0xFE，后面4个字节表示大小），通过此字段并结合ziplist.zltail属性可以实现向前遍历
encoding 1、2、5 表示当前内容的编码，最高两位表示content字段存储的数据类型（整数数组或者字节数组），其他位表示数组的长度
content 可变 表示存储的数据内容
zlend 1 压缩列表的结尾，恒为0xFF（遍历时用于判断是否到结尾）

Redis对象

Redis使用对象来表示数据库中的键和值，每次在Redis数据库中创建键值对时，都会创建两个对象，一个是键对象，一个是值对象。通过将键和值封装为对象还可以实现内存回收、内存共享（仅共享0到9999的字符串对象）、计算对象空转时间等机制。每个Redsi都由一个redisObject结构表示，其具体定义如下：

目前Redis共有字符串、列表、哈希、集合、有序集合等五种类型的Redis对象，每种类型对象至少支持两种或两种以上的编码方式，不同的编码可以在不同的使用场景优化对象使用效率。服务器在执行某些命令前会先检查给定键的类型能否执行指定的命令，而检查一个键的类型就是检查键值对象的类型。

类型 支持数据类型 编码切换条件
String int、embstr、raw int存储可转化为long型的字符串，embstr存储长度小于等于32位字符串，raw存储长度大于32位的字符串
列表 ziplist、linkedlist ziplist用于存储类型为字符串且元素数量小于512，同时满足所有字符串长度都小于64字节的列表对象，其他情况使用linkedlist编码
哈希对象 ziplist、hash ziplist用于存储key和value类型都为字符串且元素数量小于512，同时满足所有key和value字符串长度都小于64字节的哈希对象，其他情况使用hash编码
集合对象 intset、hash intset用于存储类型都为整数且元素数量小于512的集合对象，其他情况使用hash编码
有序集合 ziplist、skiplist+hash ziplist用于存储类型元素数量小于512且所有元素的长度小于64字节的有序集合对象，其他情况使用skiplist+hash编码，hash结构存储的是元素成员->元素值得映射，仅仅为了提高通过对象查询值的性能，如zscore命令
